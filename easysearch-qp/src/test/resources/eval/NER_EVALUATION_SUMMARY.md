# NER实体识别评估方案总结

## 📋 项目概述

本项目是针对搜索词实体识别（NER）系统的完整评估方案，包含：
- **流程**：搜索词处理 → 分词 → 实体识别
- **数据源**：top10w-20251010.csv (10万条用户搜索词)
- **词典库**：resources/dictionary 目录下的多个实体词典
- **实体类型**：品牌(BRAND)、型号(MODEL)、类别(CATEGORY)、成色(CONDITION)等20+种类型

## 🎯 评估指标体系

### 1. 基础指标（Core Metrics）

| 指标 | 定义 | 公式 | 目标值 |
|------|------|------|--------|
| **Precision（准确率）** | 识别结果的精确度 | TP / (TP + FP) | ≥ 0.90 |
| **Recall（召回率）** | 识别结果的完整性 | TP / (TP + FN) | ≥ 0.85 |
| **F1-Score** | 准确率和召回率的调和平均 | 2PR / (P+R) | ≥ 0.87 |
| **Accuracy（准确度）** | 整体正确率 | (TP+TN) / Total | ≥ 0.90 |

### 2. 实体级别指标（Entity-Level Metrics）

- **按实体类型统计**：每种实体类型（品牌、型号等）的 P/R/F1
- **实体边界准确率**：精确边界匹配率（目标 ≥ 80%）
- **标准化准确率**：normalizedValue 正确率（目标 ≥ 95%）

### 3. 业务指标（Business Metrics）

| 指标 | 说明 | 目标值 |
|------|------|--------|
| **Coverage（覆盖率）** | 识别出实体的查询占比 | ≥ 0.80 |
| **平均识别实体数** | 每个查询平均识别的实体数 | 1.5 - 2.5 |
| **高置信度实体占比** | 置信度 ≥ 0.8 的实体占比 | ≥ 0.75 |
| **置信度分布** | 不同置信度区间的准确率分布 | - |

### 4. 性能指标（Performance Metrics）

| 指标 | 目标值 | 优秀值 |
|------|--------|--------|
| **平均处理时间** | ≤ 50 ms | ≤ 30 ms |
| **P95 处理时间** | ≤ 100 ms | ≤ 50 ms |
| **P99 处理时间** | ≤ 200 ms | ≤ 100 ms |
| **吞吐量** | ≥ 20 qps | ≥ 30 qps |

## 📊 评估方案

### 方案1：基于标注数据的评估（Ground Truth Evaluation）

**特点**：
- ✅ 精确评估，结果可靠
- ✅ 可计算所有评估指标
- ❌ 需要人工标注，成本高
- ❌ 数据量有限

**适用场景**：
- 系统验证和效果对比
- 算法改进效果验证
- 发布前的质量评估

**执行方式**：
```bash
mvn test -Dtest=NerEvaluationTest#testWithGroundTruth
```

**数据准备**：
1. 抽样策略：
   - 高频查询（Top 10%）：100条
   - 中频查询（10%-50%）：150条
   - 低频查询（50%以下）：150条
2. 人工标注实体类型、值、位置
3. 双人标注，交叉验证

### 方案2：基于规则的自动评估（Rule-Based Evaluation）

**特点**：
- ✅ 无需标注，快速执行
- ✅ 可大规模评估
- ❌ 指标有限，准确性依赖规则
- ❌ 无法计算准确率/召回率

**适用场景**：
- 日常开发监控
- 快速效果检查
- 大规模数据分析

**执行方式**：
```bash
mvn test -Dtest=NerEvaluationTest#testAutoEvaluation
```

**评估规则**：
- 词典覆盖率检查
- 置信度分布分析
- 实体合理性检查
- 查询覆盖率统计

### 方案3：分层评估（Stratified Evaluation）

**分层维度**：

1. **按搜索频率分层**：
   - 高频查询（Top 25%）
   - 中高频查询（25%-50%）
   - 中低频查询（50%-75%）
   - 低频查询（Bottom 25%）

2. **按查询复杂度分层**：
   - 简单查询：单品牌或单型号
   - 中等查询：品牌+型号
   - 复杂查询：品牌+型号+规格

3. **按实体数量分层**：
   - 单实体、双实体、多实体查询

**执行方式**：
```bash
mvn test -Dtest=NerEvaluationTest#testStratifiedEvaluation
```

### 方案4：典型案例分析（Case Analysis）

**目的**：深入分析识别效果，发现具体问题

**案例类型**：
- 成功案例：识别效果好的查询
- 失败案例：未识别或识别错误
- 边界案例：歧义、简写等

**执行方式**：
```bash
mvn test -Dtest=NerEvaluationTest#testCaseAnalysis
```

### 方案5：实体类型覆盖率分析

**目的**：分析各类型实体的分布和覆盖情况

**执行方式**：
```bash
mvn test -Dtest=NerEvaluationTest#testEntityTypeCoverage
```

## 🗂️ 核心代码文件

### 评估框架类

1. **NerEvaluationMetrics.java** - 评估指标计算类
   - 计算 P/R/F1/Accuracy
   - 按实体类型统计
   - 置信度分布分析
   - 性能指标统计
   - 生成详细评估报告

2. **QueryEvaluationResult.java** - 单个查询的评估结果
   - 实体匹配算法
   - 边界匹配检查
   - 实体相似度计算
   - 按类型统计

3. **GroundTruthAnnotation.java** - 标注数据管理类
   - JSON格式加载/保存
   - CSV格式支持
   - 标注模板生成
   - 标注数据验证

4. **NerAccuracyTest.java** - 评估测试主类
   - 5种评估方法实现
   - 自动评估统计
   - 分层评估逻辑
   - 案例分析输出

## 📖 使用指南

### 快速开始（3步）

```bash
# 1. 运行典型案例分析（最快，1分钟内）
mvn test -Dtest=NerEvaluationTest#testCaseAnalysis

# 2. 运行自动评估（1000条数据，约2-3分钟）
mvn test -Dtest=NerEvaluationTest#testAutoEvaluation

# 3. 查看评估报告
# 报告会直接输出到控制台
```

### 完整评估流程

```
1. 准备阶段
   ├─ 确保测试数据文件存在（top10w-20251010.csv）
   ├─ 检查词典文件完整性
   └─ 配置评估参数

2. 快速评估（无需标注）
   ├─ 运行案例分析：了解基本效果
   ├─ 运行自动评估：获取覆盖率等指标
   └─ 运行分层评估：发现不同场景的问题

3. 精确评估（需要标注）
   ├─ 生成标注模板
   ├─ 人工标注数据
   ├─ 运行标注评估
   └─ 分析详细指标

4. 结果分析
   ├─ 查看评估报告
   ├─ 对比目标指标
   ├─ 识别问题点
   └─ 制定改进计划

5. 迭代优化
   ├─ 优化词典/规则/算法
   ├─ 重新评估
   └─ 对比前后效果
```

### 标注数据准备

1. **生成模板**：
```bash
mvn test -Dtest=NerEvaluationTest#testWithGroundTruth
# 会生成 src/test/resources/ner/ground_truth.json
```

2. **标注格式**（JSON）：
```json
{
  "苹果iPhone 15 Pro Max": [
    {
      "value": "苹果",
      "type": "BRAND",
      "normalizedValue": "Apple",
      "startOffset": 0,
      "endOffset": 2,
      "id": "brand_apple"
    },
    {
      "value": "iPhone 15 Pro Max",
      "type": "MODEL",
      "normalizedValue": "iPhone 15 Pro Max",
      "startOffset": 2,
      "endOffset": 20
    }
  ]
}
```

3. **标注步骤**：
   - 打开生成的 ground_truth.json
   - 检查每个实体标注是否正确
   - 修正错误，添加漏标，删除误标
   - 保存后重新运行评估

## 🔍 问题诊断与改进

### 常见问题及改进方向

| 问题 | 可能原因 | 改进方向 |
|------|---------|---------|
| **Precision低** | 误识别过多 | 优化识别规则；提高置信度阈值；改进实体归一化 |
| **Recall低** | 漏识别过多 | 扩充词典；优化分词；增加规则覆盖；降低阈值 |
| **某实体类型效果差** | 该类型词典不全 | 针对性扩充词典；优化该类型的识别规则 |
| **边界匹配率低** | 分词不准确 | 优化分词算法；调整实体边界识别逻辑 |
| **覆盖率低** | 词典覆盖不足 | 扩充词典；增加实体类型；优化识别策略 |
| **性能差** | 算法复杂度高 | 优化算法；使用缓存；索引优化；并行处理 |
| **置信度不可靠** | 置信度计算不准 | 优化置信度计算模型；引入机器学习 |

### 持续改进循环

```
评估 → 分析问题 → 制定方案 → 实施改进 → 再评估 → 对比效果
  ↑                                                      ↓
  └──────────────────────────────────────────────────────┘
```

**建议评估频率**：
- 开发期：每次重要改动后评估
- 上线前：完整评估（所有方法）
- 线上运行：
  - 每周：快速评估（自动评估）
  - 每月：完整评估（含标注评估）

## 💡 最佳实践

1. **先快后慢**：先运行快速评估了解基本情况，再决定是否需要详细评估
2. **分层分析**：不同频率、复杂度的查询效果可能差异很大，需分层分析
3. **持续迭代**：建立评估→改进→再评估的循环，持续优化
4. **重点关注**：重点关注高频查询的效果，影响最大
5. **平衡指标**：Precision和Recall需要权衡，根据业务需求调整
6. **性能监控**：在保证准确率的同时，也要关注性能指标
7. **数据积累**：逐步积累标注数据，建立评估基准
